## C1. PaySsam 콜백 서명 미검증 — 외부인 임의 결제완료 주입 가능
- **File:Line**: `src/app/api/payssam/callback/route.ts:6-71`, `src/middleware.ts:39`
- **현상**: `/api/payssam/callback`는 미들웨어 auth 면제 경로. 핸들러는 `apikey` 필드를 request body에서 꺼내지만 서버 측 `PAYSSAM_API_KEY`와 비교하지 않음. 누구든 `curl -X POST https://tuition.dminstitute.co/api/payssam/callback -d '{"bill_id":"DM-...","appr_state":"F","appr_price":"300000","appr_pay_type":"card"}'` 한 줄로 미결제 청구서를 '결제완료'로 바꿀 수 있음. `appr_state=F`이면 64~62줄에서 `tuition_payments`에 자동 insert까지 일어남.
- **원인추정**: 페이민트 연동 초기에 "일단 콜백 받기" 목적으로 auth-exempt만 넣고 서명 검증 구현을 누락. HMAC 서명 스펙이 페이민트 문서에 명시됐을 가능성 높음.
- **해결옵션**:
  1. 페이민트가 제공하는 서명 스킴 확인 후 `crypto.subtle.verify('HMAC', ...)` 또는 `crypto.timingSafeEqual`로 서명 검증 구현. 파라미터(bill_id+appr_state+appr_price+secret) SHA-256 해시 헤더 체크.
  2. 페이민트 IP 대역을 `middleware.ts`에서 화이트리스트(Cloudflare tunnel 로그에서 확인). 단, 서명 검증과 병행해야 defense-in-depth.
  3. 임시 대응: `apikey === process.env.PAYSSAM_API_KEY` 동등 비교 (timingSafeEqual 권장). 완벽하진 않지만 당장의 공개 취약은 막음.

## C2. Supabase anon key + RLS anon full CRUD — defense-in-depth 없음

- **해결옵션**:
  1. **권장**: 서버사이드 전용 `SUPABASE_SERVICE_ROLE_KEY`로 바꾸고(`src/lib/supabase.ts`에서 서버 클라이언트 분리), RLS는 `authenticated`만 허용으로 복원. 클라이언트에서 Supabase 직접 호출하는 코드가 있다면 전부 API 라우트 경유로 변경.
  2. RLS 정책에 "admin 역할" 조건 추가 + API 라우트에서 `verifyToken` 통과 여부를 Supabase JWT로 proxy. 구현 복잡도 높음.
  3. 최소 조치: `middleware.ts`에 강제 HMAC 검증(SESSION_SECRET 기본값 제거, C3과 연계)을 강화해 한 층 방어. 본질 해결은 아니지만 1안 전환까지 시간 벌이용.
